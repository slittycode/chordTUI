# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

chordTUI is a macOS terminal tool: point it at an audio file (MP3/FLAC/WAV) and it reports the
song's **key**, its **chords over time**, and the **chord progression** (triads + key at MVP).
**`PLAN.md` is the authoritative design doc** — it covers the IPC protocol (§3/§4), the phased
build plan (§7), accuracy targets and the verification gates (§9), and licensing rationale.
Read it before non-trivial changes; don't duplicate its detail here (a second source of truth
drifts).

## Commands

```bash
# Frontend (TypeScript / Bun) — run from repo root
bun test                                     # all TS tests
bun test tests/ts/music.test.ts              # one TS test file
bun test -t "roman numeral"                  # one TS test by name
bun run typecheck                            # tsc --noEmit (must be clean)
bun run dev                                   # watch-run src/index.tsx
bun run src/index.tsx analyze <file.wav>     # run the CLI directly (bin: `chordtui`, alias `chord`)
bun link                                      # install `chordtui` (+ `chord`) globally on PATH (~/.bun/bin)

# Python engine — also run from repo root (uv resolves the engine/ project)
uv run --project engine pytest               # all Python tests (incl. the blocking librosa accuracy gate)
uv run --project engine pytest tests/py/test_librosa_engine.py -k "test_key_is_c_major"
uv run --project engine ruff check engine tests/py        # lint engine + tests
uv run --project engine python tests/fixtures/make_fixtures.py   # regenerate the committed accuracy fixtures
```

`uv run --project engine pytest` works from the repo root because `engine/pyproject.toml` sets
`testpaths = ["../tests/py"]` — the test files live *outside* the `engine/` package dir, in
`tests/py/`. Those tests run the real librosa engine end-to-end (sys.executable is the py3.9 venv
that has librosa). `test_librosa_engine.py` synthesizes its WAV at runtime; the **accuracy gate**
(`tests/py/accuracy.py` + `test_accuracy.py`) runs over the **committed CC0 synthetic fixtures** in
`tests/fixtures/audio/` (generated by `make_fixtures.py`), blocking at key ≥ 0.75 / chord ≥ 0.55.
The btc fidelity/engine tests are `pytest.importorskip("torch")`-skipped in CI; run them with the
btc venv: `engine/.venv-btc/bin/python -m pytest tests/py/test_btc_fidelity.py`. The fixtures are
synthetic harmonic triads (no copyrighted audio); real-audio accuracy is measured separately via
`tools/eval_guitarset.py` on GuitarSet (see `docs/probe-matrix.md §8`).

## Architecture

**Polyglot split coupled only by a versioned JSON contract.** A TypeScript/Bun + OpenTUI
frontend drives a Python analysis sidecar over `Bun.spawn`. The contract is the *sole* coupling,
so the two halves are independently testable and a future engine can drop in without frontend
changes. Read the file headers of `src/core/types.ts` and `src/core/engine.ts` — they document
their own invariants thoroughly.

### The contract is a three-way lockstep (biggest edit hazard)

These four must move together, or the round-trip tests (`tests/ts/contract.test.ts`,
`tests/py/test_schema.py`) fail:

1. `src/core/types.ts` — the TS types (the canonical shape).
2. `src/core/validate.ts` — a **hand-rolled** runtime validator (no zod/ajv). Enforces strict
   "no unexpected property" checks mirroring JSON Schema `additionalProperties: false`.
3. `engine/schema.json` — JSON Schema for the `Analysis` payload.
4. `engine/engine-info.schema.json` — JSON Schema for the cheap `engine-info` probe.

Change one field → change all four. Both sides enforce strict unknown-key rejection.

### IPC protocol (see PLAN.md §4 for the full spec)

- **Invocation:** array-form `Bun.spawn`, never `sh -c` (the file path is untrusted input).
- **stdout:** exactly one JSON document — an `Analysis` or `{ error: EngineError }`. Buffered
  fully (capped, never truncated), then parsed + validated.
- **stderr:** NDJSON, one progress/log event per line, streamed incrementally.
- **exit codes:** 0 ok · 2 bad input · 3 engine-unavailable (no stdout → route to `doctor`) ·
  4 analysis/internal. `src/core/engine.ts` owns the cancel ladder (AbortSignal/timeout →
  SIGTERM → grace → SIGKILL); it mirrors `~/code/projects/tempo/src/daemon.ts`, deliberately
  **not** tempo's `executor.ts`.

### Tiered engines, capability-gated UI

Engines speak the *same* contract, differentiated only by `engineCapabilities[]`, `vocabulary`,
and `confidenceKind` — **never by sniffing the engine name**: librosa (ISC, always installs;
triads-only preview + fallback) and **btc** (MIT — vendored BTC-ISMIR19 in `engine/vendor/btc`,
the opt-in default accuracy tier: extended chords + chord-derived key, ~80% accuracy). essentia
(AGPL) is reserved/unimplemented. **btc is fully permissive — no NonCommercial consent.** A UI
panel renders **iff** its capability is present; a nullable field that is `null` means "engine
couldn't compute" → the sub-feature is removed, **never faked**. A *missing* required field is a
malformed-output error, not a null.

### Pure music theory stays TS-side

`src/core/music.ts` derives enharmonic spelling and **in-key** roman numerals from
`(key, root, quality)` — these are never sent over the wire. Out-of-key / unknown chords get
**no numeral** (omit, never invent). Pure functions, unit-tested without a renderer.

## Invariants & gotchas

1. **`chords[]` is gap-free and contiguous** over `[0, durationSec]`: `chords[i].end ===
   chords[i+1].start`. Silence is an explicit `"N"` (no-chord) segment, never a gap. The
   validator enforces this (JSON Schema can't).
2. **`process.exitCode = …`, never `process.exit()`** in `src/index.tsx` — `process.exit()` can
   drop a buffered `--json` stdout write before it flushes.
3. **Python producers serialize with `allow_nan=False`** (`engine/protocol.py:dumps`) — a bare
   `NaN`/`Infinity` token is invalid JSON for Bun and slips past schema numeric checks; fail loud.
4. **Mock-sidecar gate** (`src/cli/commands.ts:gateMock`): with no real engine, the bundled mock
   emits contract-conformant **fake** data. It runs only in an interactive TTY (no `--json`) or
   when `$CHORDTUI_SIDECAR` is set; piped / `--json` refuses with exit 3 so sample chords never
   leak into a script or bug report. Tests bypass this via the injected `analyzeBase` /
   `engineInfoBase` seam, not env vars.
5. **Two venvs.** The clean-core `engine/.venv` (py3.9, librosa) is what `uv sync`/CI use. The
   **btc** tier installs into a SEPARATE `engine/.venv-btc` (py3.11 + torch + librosa) so torch
   never enters the clean-core lock; `resolveEngine` prefers `.venv-btc` when present (it serves
   both engines). `chord setup` builds it via `uv venv` + `uv pip install torch …` — NOT a uv
   extra, so default `uv sync` stays lean. btc tests are `pytest.importorskip("torch")`-skipped in
   CI; run them with `engine/.venv-btc/bin/python -m pytest`.
6. **Contract version is major-1** (`1.x.y`); the frontend rejects a mismatched major.
7. **librosa no-chord is gated on raw-signal RMS** (`engine/engines/librosa_engine.py`
   `_SILENCE_RMS`), *not* the chroma norm: `chroma_cqt` normalizes every frame to unit-max, so a
   silent frame looks like a full chord — the old `_SILENCE_NORM` chroma check was dead code.
   Silence ("N") is detected from `librosa.feature.rms` on the same hop grid.
8. **Result cache** (`src/core/cache.ts`): keyed on `(sha256(file bytes), engine)` under
   `~/.cache/chordtui/results/`; `analyzeWithCache` is the shared CLI+TUI seam. It **never caches
   the mock** (`isMock`) and **re-validates every entry on read** (`cacheGet` → `validateAnalysis`,
   so a wrong-major/corrupt entry is ignored → recompute). `--no-cache` bypasses it.
9. **Env seams** (read lazily, override for tests): `$CHORDTUI_ENGINE_DIR` / `$CHORDTUI_PYTHON` /
   `$CHORDTUI_SIDECAR` (engineResolve), `$CHORDTUI_CACHE_DIR` (cache.ts).

## Current state vs PLAN.md

The plan describes the full system; the repo is partway through it. Notably:

1. **Built:** the contract (types/validate/schemas), `engine/mock_sidecar.py`, the librosa engine,
   the **btc engine** (`engine/engines/btc_engine.py` + vendored `engine/vendor/btc`, MIT) — the
   default accuracy tier: BTC-ISMIR19 extended chords + chord-derived key, `engine.ts` spawn seam,
   `src/core/cache.ts` (per-engine result cache, with engine-info staleness), the CLI (`analyze` /
   `engine-info` / `doctor` / `setup`), and the **interactive TUI** (`src/hooks/useAnalysis.ts`
   preview→upgrade machine + pure reducer, `src/core/panels.ts` gating, `src/components/`). Plus
   the **accuracy gates**: librosa synthetic-fixture floor (`tests/py/test_accuracy.py`, blocking);
   the btc **fidelity gate** (`tests/py/test_btc_fidelity.py` — byte-equal vs BTC's reference, so
   we inherit its published ~80.8% MIREX WCSR); and the GuitarSet real-audio measurement
   (`tools/eval_guitarset.py` → 0.76 chord / 0.84 key, `docs/probe-matrix.md §8`).
2. **Not built yet** (don't go hunting): the **essentia** engine module (`analyze.py` returns
   `engine_unavailable`). madmom has been **retired** (BTC replaced it — no NonCommercial models,
   no consent flow; `consent.ts` is gone). btc ships its module + vendored weights in-repo but its
   heavy package (torch) is **not installed by default**: `chord setup` installs it into
   `engine/.venv-btc`; until then btc reports `engine_unavailable` and the TUI falls back to the
   librosa preview.
3. **Behavior to know:** `resolveEngine` (`src/core/engineResolve.ts`, shared by CLI + hook)
   prefers `engine/.venv-btc` when present. Results are cached per `(sha256(file), engine)` via
   `analyzeWithCache`; never cached for the mock; a cached btc entry is invalidated when the
   engine's version/`modelVersions` change. `chord setup` installs the librosa clean core
   (`uv sync`) and, by default, btc (`--no-btc` to skip) — no consent gate (btc is MIT). `chord
   doctor` shows a per-engine table whose **working** column means a processor actually ran on a
   test WAV (`engine/selftest.py`). The CLI default engine and the TUI auto-upgrade pick btc when
   installed, else librosa.
